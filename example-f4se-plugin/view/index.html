null return - when and why
Both read functions return null (never throw) when:
• Plugin not in the active load order
• Form ID does not exist in that plugin
• Form exists but wrong type (not TESGlobal, or script not attached)
• Property not declared Auto or name mismatch
• Papyrus VM not yet ready (called before kPostLoadGame)

Always guard: if (val === null) { /* missing or not loaded */ }
C++ → JS - Invoke() and InteropCall()
api->Invoke(view, "fnName(args)") - evaluates a JS expression. Higher overhead; use for one-shot calls like init and infrequent UI updates.
api->InteropCall(view, "fnName", jsonArg) - calls a named window function directly. Lower overhead; use for per-frame or repeated data pushes. Both auto-convert ANSI game strings to UTF-8.

The focus badge below is driven by C++ calling Invoke(view, "updateFocusLabel('...')").

JS → C++ - BindUIEvent()
api->BindUIEvent(view, "eventName", callback) exposes a named function on window that JS calls like any other function.
The callback fires on the game thread - RE:: access is safe directly inside it. No AddTask needed.

RegisterJSListener() - the original JS to C++ callback
Same shape as BindUIEvent: exposes a named function on window that JS calls like any other function. It also dispatches onto the game thread today, so RE:: access is safe directly inside it - no AddTask needed.
The difference is contract, not behavior: BindUIEvent (V4) makes the game-thread guarantee part of its actual API contract, while RegisterJSListener only happens to work that way with the current backend. Prefer BindUIEvent in new code; existing RegisterJSListener code is not broken and needs no changes.

Always register inside OnDomReadyCallback - not before DOM ready.
View lifecycle - call-timing rules
RequestPluginAPI<IVPrismaUI4>() → on kGameDataReady
CreateView() → on kPostLoadGame or kNewGame
RegisterJSListener() → inside OnDomReadyCallback only
Hide(view) → immediately after CreateView unless you want it visible
IsValid(view) → always check after a game reload before using a stored handle

Violating these timings causes null crashes or silent no-ops.

Quick Start - 30 Seconds
1. Press F3 in-game to open this panel.
2. Go to Papyrus Bridge tab.
3. Try Read Global with any mod's plugin name and form ID (e.g., MyMod.esp + 800).
4. Check the result below the button. If it says null, the mod or form doesn't exist. If it shows a number, you've read a Papyrus global successfully!
Papyrus Bridge - What It Does
Read/Write Globals: Get or set the value of a TESGlobal form from any loaded mod. Type the plugin name and the form's local hex ID (without the file-index byte). The change happens instantly in memory.

Read/Write Properties: Get or set the value of an Auto property from a Papyrus script attached to any form. Requires all fields: plugin, form ID, script name, and property name.

Direct Writes Note: Writing to properties directly updates the Papyrus VM's instance variable, but it does not execute Papyrus function code automatically. If you need script code to execute when a property changes, poll the value in Papyrus (e.g. via RegisterForSingleUpdate) or use MCM callbacks.

Common Patterns
1. Check if a mod is loaded: Call getGlobal() with any form from that mod. If it returns null, the mod is not loaded or the form doesn't exist.

2. Toggle a feature: Read a global (get the current value), then write it back inverted: newValue = currentValue === 0 ? 1 : 0.

3. Use quest forms: Always store Papyrus properties on quest forms. They persist across cell changes and are the most reliable host for script properties.

4. Handle null: Always guard your reads: if (val === null) { console.log('form not found'); }
API Timing Rules (Advanced)
window.prisma is injected by PrismaUI_F4 before your view loads. It's safe to use in your init() function or later.

await prisma.getProperty() and await prisma.getGlobal() return Promises. Always use await or .then().

prisma.setGlobal() is fire-and-forget - it doesn't return a result. Just send and continue.
